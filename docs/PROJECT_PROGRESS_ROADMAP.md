# Wang-Detective 项目进度与后续路线

## 1. 最初诉求回顾

最初目标是把原始 King-Detective/OCI 管理项目下载到本地，全面检查、修复、优化，并参考同类 R-Bot / java_oci_manage 方向补齐更实用的功能。新版本代码需要独立上传到 `tony-wang1990/Wang-Detective` 仓库。

优先级大致分为三层：

1. 先修复会影响部署、启动、更新、数据库、WebSocket、分页和健康检查的稳定性问题。
2. 再补同类项目中最实用的运维入口能力，例如 Web SSH、SFTP、批量命令、主机资产库。
3. 后续继续扩展 OCI Object Storage、Email Delivery、Always Free 风险看板、多云资产同步、端口转发和更细权限控制。

### 2.3 SSH 主机资产库

- 新增 `ops_ssh_host` 表。

- 新增 `OPS_SSH_SECRET_KEY`，建议生产环境固定配置。
- 运维页支持选择保存主机，不需要每次手动输入 SSH 信息。
- 批量命令支持对全部保存主机执行。

### 2.4 运维审计

- 补齐 `audit_log` 的 Mapper 和 Service 实现。

- 运维页新增“操作审计”标签页。
- 已记录的操作包括：
  - SSH 主机新增、更新、删除。

## 3. 相比原始项目的主要变化

原始项目核心偏向 OCI API 管理和 Telegram Bot 操作，已经具备实例、开机任务、VNC、引导卷、VCN、安全规则、Cloudflare DNS、备份恢复、MFA、通知等能力。

当前增强版主要补齐了三块短板：

- 部署可靠性：更新、持久化、数据库迁移、健康检查和诊断接口更稳。
- 实际运维入口：从“只管 OCI 资源”扩展到“能直接登录机器、传文件、执行命令”。
- 安全可追溯：保存主机凭据加密、关键运维操作有审计记录。

## 5. Telegram Bot 后续可增加的菜单

Telegram Bot 仍有很大扩展空间，建议不要一次性塞太多，按“高频、低风险、可回滚”顺序做。

### 5.1 运维菜单

- 每日资源摘要。

### 5.4 备份菜单
